# 使用 SmolLM3 进行监督微调

监督微调 (Supervised Fine-Tuning, SFT) 是指令微调的基石——它是我们如何将基础语言模型转变为遵循指令的助手。在本节中，你将学习使用真实世界数据集和生产就绪工具微调 **SmolLM3**。

## 什么是监督微调？

SFT 是在特定任务数据集上使用标记示例继续训练预训练模型的过程。可以把它想象成专业教育：

- 预训练教会模型一般语言理解（就像学习阅读）。
- 监督微调教会特定技能和行为（就像学习完成特定任务）。

SFT 背后的关键见解是，我们不是从头开始教模型新知识。相反，我们是在**重塑现有知识的应用方式**。预训练模型已经理解语言、语法，并吸收了大量的事实信息。SFT 将这种通用能力聚焦于特定的应用模式、响应风格和特定任务的要求。

这种方法之所以有效，是因为它利用了预训练期间学到的丰富表示，同时需要的计算资源远少于从头开始训练。模型学会识别指令模式、保持对话上下文、遵循安全指南，并以所需格式生成响应。

> [!TIP]
> 在开始 SFT 之前，请考虑使用现有的指令微调模型和精心设计的提示词是否足以满足你的用例。SFT 涉及大量的计算资源和工程工作，因此只有在提示现有模型不足时才应追求。在 [Hugging Face 大语言模型课程](https://huggingface.co/learn/llm-course/en/chapter11/3)中了解更多关于这个决策过程的信息。

### SmolLM3 的 SFT 之旅

SmolLM3 的指令遵循能力来自复杂的 SFT 过程：

1. **基础模型** (`SmolLM3-3B-Base`)：在 11T 词元的通用文本上训练
2. **SFT 训练**：在精心策划的指令数据集（包括 SmolTalk2）上微调
3. **偏好对齐**：使用 APO（锚定偏好优化）等技术进一步优化

这种多阶段方法创造了一个既有知识又有帮助的模型。

## 为什么 SFT 有效：背后的科学

SFT 之所以有效，是因为它利用了预训练期间学到的丰富表示，同时调整了模型的行为模式。在 SFT 期间，模型的参数通过在特定任务示例上的梯度下降进行微调，导致模型处理和生成文本的方式发生细微但重要的变化。

具体来说，这个过程通过几个关键机制起作用：

**行为适应**：模型学会识别指令模式并做出适当响应。这涉及更新注意力机制以关注语言中的指令提示，并调整输出分布以偏向所需的响应。研究表明，指令微调主要影响模型的表面行为，而不是其底层知识 [(Wei et al., 2021)](https://huggingface.co/papers/2109.01652)。

**任务专业化**：模型不是学习全新的概念，而是学习在特定上下文中应用其现有知识。这就是为什么 SFT 比预训练更高效——我们在完善现有能力，而不是从头构建它们。研究表明，大部分事实知识来自预训练，而 SFT 教会模型如何适当地格式化和呈现这些知识 [(Ouyang et al., 2022)](https://huggingface.co/papers/2203.02155)。

**安全对齐**：通过接触精心策划的示例，模型学会变得更有帮助、无害和诚实。这涉及学习在各种情况下该说什么和不该说什么。这种方法的有效性已在 InstructGPT [(Ouyang et al., 2022)](https://huggingface.co/papers/2203.02155) 和 Constitutional AI [(Bai et al., 2022)](https://huggingface.co/papers/2204.05862) 等工作中得到证明。

> [!TIP]
> SFT 不教授新事实——它教授新行为。模型已经从预训练中了解了世界；SFT 教它如何使用这些知识成为一个有帮助的助手。

数学基础涉及最小化模型预测与训练数据集中目标响应之间的交叉熵损失。这个过程逐渐改变模型的概率分布，以偏向训练示例中演示的响应类型。

## 何时使用监督微调

关键问题是："我的用例是否需要与通用对话显著不同的行为？"如果是，SFT 可能是有益的。

决策框架：使用此清单来确定 SFT 是否适合你的项目：
- 你是否尝试过使用现有指令微调模型的提示词工程？
- 你是否需要提示无法实现的一致输出格式？
- 你的领域是否足够专业化，以至于通用模型难以应对？
- 你是否拥有高质量的训练数据（至少 1,000 个示例）？
- 你是否拥有训练和评估的计算资源？

如果你对大多数问题回答"是"，那么 SFT 可能值得追求。

## SFT 过程

现在让我们继续讨论 SFT 本身的过程。SFT 过程遵循系统性方法，确保高质量结果：

### 1. 数据集准备和选择

训练数据的质量是成功 SFT 的最关键因素。与预训练（数量通常最重要）不同，SFT 优先考虑质量和相关性。你的数据集应包含输入-输出对，准确展示你希望模型学习的行为。

**选择合适的数据集**：
- SmolTalk2：用于训练 SmolLM3 的数据集，包含高质量的指令-响应对。
- 特定领域数据集：用于专业应用（医疗、法律、技术）。
- 自定义数据集：你自己为特定用例策划的示例。

每个训练示例应包含：
1. **输入提示词**：用户的指令或问题
2. **预期响应**：理想的助手响应
3. **上下文**（可选）：所需的任何附加信息


> [!TIP]
> 数据集大小指南：
> - 最小值：1,000 个高质量示例用于基本微调。
> - 推荐值：10,000+ 个示例以实现稳健性能。
> - 质量优于数量：1,000 个精心策划的示例通常优于 10,000 个平庸的示例。
>
> 请记住：你的模型将学会模仿训练数据中的模式，因此请在数据策划上投入时间。

### 2. 环境设置和配置

要设置 SFT 环境，我们需要高级计算资源。我们有三个主要选项：

1. **本地 GPU**：如果你足够幸运能够访问具有（至少 16GB VRAM）的 GPU，你可以在本地训练模型！
2. **Hugging Face Jobs**：如果你没有 GPU 并且不想使用云提供商，你可以使用 Hugging Face Jobs！我们将在[下一节](./5)中详细介绍这一点。
3. **笔记本 GPU**：如果你喜欢使用 Google Colab 等笔记本提供商，你可以使用他们的 GPU！
4. **云 GPU**：如果你想控制你的计算资源，你可以使用 AWS、GCP 或 Azure 等云提供商。

就硬件要求而言，你需要一个至少具有 16GB VRAM 的 GPU，例如 Nvidia RTX 4080 或 A10G。

### 3. 训练配置

选择正确的超参数对于成功的 SFT 至关重要。目标是找到模型有效学习而不会过拟合或变得不稳定的最佳点。以下是每个参数的详细分解以及如何选择它们：

**关键超参数**：

**学习率** (5e-5 to 1e-4)：控制每次更新模型权重变化的程度
- 从 5e-5 开始用于 SmolLM3；这是保守且稳定的。
- 太高：模型变得不稳定；损失振荡或爆炸。
- 太低：模型学习非常缓慢，可能无法在合理时间内收敛。

**批次大小** (4-16)：同时处理的示例数量
- 较大批次：更稳定的梯度，但需要更多 GPU 内存。
- 较小批次：更少的内存使用，但梯度更嘈杂。
- 使用梯度累积来实现更大的有效批次大小。

**最大序列长度** (2048-4096)：每个训练示例的最大词元数
- 较长序列：可以处理更复杂的对话。
- 较短序列：更快的训练，更少的内存使用。
- 匹配你的用例：使用目标对话的典型长度。

**训练步数** (1000-5000)：参数更新的总次数
- 取决于数据集大小：更多数据通常需要更多步数。
- 监控验证损失：当它停止改善时停止。
- 经验法则：遍历数据集三到五个周期。

**预热步数** (总数的 10%)：开始时的学习率逐渐增加
- 防止早期不稳定：帮助模型逐渐适应。
- 典型范围：大多数 SFT 任务为 100-500 步。

> [!TIP]
> SmolLM3 的超参数起点：
>
> 要引导你的训练，你可以使用以下超参数：
>
> **学习率**：
>
> ```python

### 4. 监控和评估

有效的监控对于成功的 SFT 至关重要。与预训练（你主要观察损失下降）不同，SFT 需要仔细关注定量指标和定性输出。目标是确保你的模型正在学习所需的行为，而不会过拟合或产生不需要的模式。

**要监控的关键指标**：

**训练损失**：应该稳定下降，但不要太快
- 健康模式：平滑、逐渐下降。
- 警告信号：突然尖峰、振荡或平台期。
- 典型范围：从大约 2-4 开始，应降至 0.5-1.5。

**验证损失**：防止过拟合的最重要指标
- 应该跟踪训练损失：小间隙表示良好的泛化。
- 增长间隙：过拟合迹象；模型可能正在记忆训练数据。
- 用于早停：当验证损失停止改善时停止训练。

**样本输出**：定期定性检查至关重要
- 生成响应：在训练期间在保留的提示词上测试模型。
- 检查格式一致性：确保模型遵循所需的响应模式。
- 监控退化：注意重复或无意义的输出。

**资源使用**：跟踪 GPU 内存和训练速度
- 内存尖峰：可能表示批次大小太大。
- 训练缓慢：可能表明数据加载或处理效率低下。

## 理解 SFT 中的损失模式

训练损失通常遵循三个不同的阶段，如 [Hugging Face 大语言模型课程](https://huggingface.co/learn/llm-course/en/chapter11/3)中的示例所示：

<div class="flex justify-center">
<img src="https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/nlp_course_sft_loss_graphic.png" alt="SFT Training Progress" width="80%"/>
</div>

1. **初始急剧下降**：快速适应新数据分布
2. **逐渐稳定**：学习率减慢，模型微调
3. **收敛**：损失值稳定，表明训练完成

**健康的训练模式**：成功训练的关键指标是训练损失和验证损失之间的间隙较小，表明模型正在学习可泛化的模式，而不是记忆特定示例。

## 需要注意的警告信号

损失曲线中的几种模式可以指示潜在问题：

### 过拟合模式

<div class="flex justify-center">
<img src="https://huggingface.co/datasets/smol-course/images/resolve/main/sft_loss_1.png" alt="SFT Overfitting Pattern" width="80%"/>
</div>

如果验证损失增加而训练损失继续下降，你的模型正在过拟合。考虑：
- 减少训练步数或周期
- 增加数据集大小或多样性
- 添加正则化技术
- 使用基于验证损失的早停

### 欠拟合模式

<div class="flex justify-center">
<img src="https://huggingface.co/datasets/huggingface/documentation-images/resolve/main/sft_loss_2.png" alt="SFT Underfitting Pattern" width="80%"/>
</div>

如果损失没有显示显著改善，模型可能：
- 学习太慢（尝试增加学习率）
- 难以处理任务复杂性（检查数据质量）
- 达到架构限制（考虑不同的模型大小）

### 潜在的记忆

<div class="flex justify-center">
<img src="https://huggingface.co/datasets/smol-course/images/resolve/main/sft_loss_3.png" alt="SFT Memorization Pattern" width="80%"/>
</div>

极低的损失值可能表明记忆而不是学习。如果出现以下情况，这令人担忧：
- 模型在新的类似示例上表现不佳
- 输出缺乏多样性或创造力
- 响应与训练示例太相似

## 记录的指标

在训练和评估时，我们记录以下奖励指标：

* `global_step`：到目前为止采取的优化器步骤总数。
* `epoch`：基于数据集迭代的当前周期数。
* `num_tokens`：到目前为止处理的词元总数。
* `loss`：在当前记录间隔内对非掩码词元计算的平均交叉熵损失。
* `entropy`：模型预测的词元分布在非掩码词元上的平均熵。
* `mean_token_accuracy`：模型的 top-1 预测与真实词元匹配的非掩码词元的比例。
* `learning_rate`：当前学习率，如果使用调度器，它可能会动态变化。
* `grad_norm`：梯度的 L2 范数，在梯度裁剪之前计算。

## 预期的数据集类型和格式

SFT 支持[语言建模](https://huggingface.co/docs/trl/dataset_formats#language-modeling)和[提示词-补全]([dataset_formats](https://huggingface.co/docs/trl/dataset_formats)#prompt-completion)数据集。[`SFTTrainer`] 与[标准]([dataset_formats](https://huggingface.co/docs/trl/dataset_formats)#standard)和[对话]([dataset_formats](https://huggingface.co/docs/trl/dataset_formats)#conversational)数据集格式兼容。当提供对话数据集时，训练器将自动对数据集应用对话模板。

```python
# 标准语言建模
{"text": "The sky is blue."}

# 对话语言建模
{"messages": [{"role": "user", "content": "What color is the sky?"},
              {"role": "assistant", "content": "It is blue."}]}

# 标准提示词-补全
{"prompt": "The sky is",
 "completion": " blue."}

# 对话提示词-补全
{"prompt": [{"role": "user", "content": "What color is the sky?"}],
 "completion": [{"role": "assistant", "content": "It is blue."}]}
```

如果你的数据集不是这些格式之一，你可以预处理它以将其转换为预期格式。以下是 [FreedomIntelligence/medical-o1-reasoning-SFT](https://huggingface.co/datasets/FreedomIntelligence/medical-o1-reasoning-SFT) 数据集的示例：

```python
from datasets import load_dataset

dataset = load_dataset("FreedomIntelligence/medical-o1-reasoning-SFT", "en")

def preprocess_function(example):
    return {
        "prompt": [{"role": "user", "content": example["Question"]}],
        "completion": [
            {"role": "assistant", "content": f"<think>{example['Complex_CoT']}</think>{example['Response']}"}
        ],
    }

dataset = dataset.map(preprocess_function, remove_columns=["Question", "Response", "Complex_CoT"])
print(next(iter(dataset["train"])))
```

## 训练中的对话模板

我们将在训练的上下文中简要回到对话模板。在训练期间正确使用对话模板对于模型性能至关重要。以下是关键考虑事项和最佳实践：

### 预处理和分词

在训练期间，每个示例都应包含一个 **text 字段**或一个 **(prompt, completion)** 对，具体取决于数据集格式。有关预期格式的更多详细信息，请参见[数据集格式](https://huggingface.co/docs/trl/dataset_formats)。
[`SFTTrainer`](https://huggingface.co/docs/trl/sft_trainer) 使用模型的分词器对每个输入进行分词。如果提示词和补全分别提供，它们在分词之前会被连接起来。

### 计算损失

![sft_figure](https://huggingface.co/datasets/trl-lib/documentation-images/resolve/main/sft_figure.png)

SFT 中使用的损失是**词元级交叉熵损失**，定义为：

$$
\mathcal{L}_{\text{SFT}}(\theta) = - \sum_{t=1}^{T} \log p_\theta(y_t \mid y_{<t}),
$$

其中 \\( y_t \\) 是时间步 \\( t \\) 的目标词元，模型被训练来根据前面的词元预测下一个词元。在实践中，填充词元在损失计算期间被掩码掉。

## 使用 TRL (Transformer Reinforcement Learning) 进行监督微调

**TRL** 是训练语言模型的首选工具包，专门为指令微调和对齐而构建。这是我们在整个课程中将使用的工具。

### 为什么选择 TRL？

- 生产就绪：被主要组织和研究实验室使用。
- 全面：支持 SFT、DPO、ORPO、PPO 和更高级的技术。
- 高效：针对内存使用和训练速度进行了优化。
- 灵活：适用于任何 Hugging Face 模型。
- CLI 支持：用于可扩展训练工作流的命令行工具。

### 关键组件

- **SFTTrainer**：监督微调的核心类
- **SFTConfig**：训练参数的配置管理
- **CLI 工具**：用于生产工作流的命令行界面
- **集成**：与 Hugging Face Hub、Trackio、Weights & Biases 等无缝集成

### TRL 的架构

TRL 建立在 Hugging Face 生态系统之上：
- Transformers：模型加载和推理。
- Datasets：数据处理和管理。
- Accelerate：分布式训练和优化。
- PEFT：参数高效微调（LoRA、QLoRA）。

这种集成方法意味着你可以在使用最先进的训练技术的同时获得 Hugging Face 生态系统的所有好处。

> [!TIP]
> TRL 与其他训练库的比较：
> - TRL：专门用于 LLM 训练，为指令微调而构建。
> - Transformers Trainer：通用，适用于基本微调。
> - DeepSpeed：专注于大规模分布式训练。
> - Accelerate：提供低级分布式训练原语。
>
> TRL 为 SFT 提供了易用性和高级功能的最佳平衡。有关训练方法的更多详细信息，请参见 [Hugging Face 大语言模型课程](https://huggingface.co/learn/llm-course/en/chapter11/3)。

## 实践：你的第一次 SmolLM3 微调

准备好将理论付诸实践了吗？以下是你将在练习中构建的内容预览。你可以使用 Python 或 CLI 方法：

<hfoptions id="first_fine_tune">
<hfoption id="python">

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from trl import SFTTrainer, SFTConfig
from datasets import load_dataset
import trackio as wandb

# 初始化实验跟踪
wandb.init(project="smollm3-sft", name="my-first-sft-run")

# 加载 SmolLM3 基础模型
model = AutoModelForCausalLM.from_pretrained("HuggingFaceTB/SmolLM3-3B-Base")
tokenizer = AutoTokenizer.from_pretrained("HuggingFaceTB/SmolLM3-3B-Base")

# 加载 SmolTalk2 数据集
dataset = load_dataset("HuggingFaceTB/smoltalk2_everyday_convs_think")

# 配置训练并集成 Trackio
config = SFTConfig(
    output_dir="./smollm3-finetuned",
    per_device_train_batch_size=4,
    learning_rate=5e-5,
    max_steps=1000,
    report_to="trackio",  # 启用 Trackio 记录
)

# 训练！
trainer = SFTTrainer(
    model=model,
    train_dataset=dataset["train"],
    args=config,
)
trainer.train()
```

</hfoption>

    --save_steps 200 \
    --report_to trackio \

    --hub_model_id your-username/smollm3-custom
